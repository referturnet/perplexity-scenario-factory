error-handli# Pattern: Retry Logic

## Описание

Паттерн для обработки временных сбоев с автоматическими повторными попытками.

## Проблема

Веб-операции могут временно завершаться неудачей из-за:
- Сетевых задержек
- Временной недоступности сервера
- Таймаутов
- Неполной загрузки элементов
- Race conditions

## Решение

Реализовать механизм повторных попыток с экспоненциальной задержкой и ограничением количества попыток.

---

## Реализация

### Базовая структура

```markdown
Функция retry_operation:
  max_attempts = 3
  delay = 1 секунда
  
  Для каждой попытки (1..max_attempts):
    Попробовать:
      выполнить операцию
      если успешно -> вернуть результат
    При ошибке:
      если попытка < max_attempts:
        подождать delay секунд
        delay = delay * 2  // экспоненциальная задержка
        продолжить цикл
      иначе:
        вернуть ошибку
```

### Псевдокод для Perplexity Agent

```markdown
## Retry для навигации

max_attempts = 3
current_attempt = 0
success = false

Пока current_attempt < max_attempts И НЕ success:
  current_attempt += 1
  
  Попробовать:
    1. Navigate к URL
    2. Wait 2 секунды
    3. Проверить наличие expected_element через read_page
    4. Если элемент найден:
       success = true
    5. Иначе:
       Если current_attempt < max_attempts:
         Wait (current_attempt * 2) секунд
         
Если НЕ success:
  Логировать ошибку
  Вернуть failure
```

## Типы retry стратегий

### 1. Fixed Delay (Фиксированная задержка)

```markdown
Задержка между попытками постоянная

Попытка 1 -> ошибка -> wait 2s
Попытка 2 -> ошибка -> wait 2s  
Попытка 3 -> ошибка -> fail

✅ Просто
❌ Может создавать постоянную нагрузку
```

### 2. Exponential Backoff (Экспоненциальная задержка)

```markdown
Задержка увеличивается с каждой попыткой

Попытка 1 -> ошибка -> wait 1s
Попытка 2 -> ошибка -> wait 2s
Попытка 3 -> ошибка -> wait 4s
Попытка 4 -> ошибка -> fail

✅ Снижает нагрузку
✅ Дает время восстановиться серверу
❌ Длительнее выполнение
```

### 3. Exponential Backoff + Jitter (С случайным сдвигом)

```markdown
Экспоненциальная задержка + случайное отклонение

Попытка 1 -> wait random(0.5-1.5)s
Попытка 2 -> wait random(1-3)s
Попытка 3 -> wait random(2-6)s

✅ Избегает "thundering herd"
✅ Оптимален для параллельных операций
```

## Примеры использования

### Пример 1: Retry для клика по элементу

```markdown
## Задача: Кликнуть по кнопке, которая может загружаться динамически

max_attempts = 3

Для попытки 1..3:
  1. Использовать read_page для поиска кнопки
  2. Если найдена:
     - Кликнуть по ref
     - Проверить результат
     - Если успешно -> выход
  3. Если не найдена:
     - Wait (попытка * 2) секунд
     - Обновить страницу через screenshot
     - Повторить
     
Если все попытки исчерпаны:
  Логировать: "Элемент не найден после 3 попыток"
  Сделать screenshot для отладки
```

### Пример 2: Retry для извлечения данных

```markdown
## Задача: Извлечь текст, который появляется после AJAX запроса

max_attempts = 4
delay = 1

Для каждой попытки:
  1. Использовать get_page_text с query="нужные данные"
  2. Если текст содержит искомую информацию:
     - Сохранить результат
     - Выход из цикла
  3. Иначе:
     - Wait delay секунд
     - delay = delay * 1.5
     - Продолжить
```

### Пример 3: Retry для навигации

```markdown
## Задача: Перейти на страницу, проверив успешность

max_attempts = 3
target_url = "https://example.com/page"

Для попытки 1..3:
  1. navigate(target_url)
  2. wait 3 секунды
  3. screenshot для проверки URL
  4. read_page для проверки ожидаемого контента
  5. Если URL корректный И контент найден:
     - Успех
  6. Иначе:
     - Если попытка < 3:
       - wait (попытка * 3) секунд
       - Попытка снова
```

## Когда использовать

### ✅ Используй retry для:

1. **Сетевых операций**
   - Загрузка страниц
   - AJAX запросы
   - API вызовы

2. **Динамического контента**
   - Элементы, загружаемые асинхронно
   - Модальные окна
   - Lazy-loaded изображения

3. **Нестабильных элементов**
   - Элементы, которые могут появляться с задержкой
   - Элементы в процессе анимации

4. **Временных сбоев**
   - 502/503 ошибки сервера
   - Timeouts
   - Rate limiting

### ❌ НЕ используй retry для:

1. **Постоянных ошибок**
   - 404 страница не найдена
   - 401 не авторизован
   - Отсутствующие элементы (by design)

2. **Операций с побочными эффектами**
   - Отправка форм (может создать дубликаты)
   - Платежные транзакции
   - Удаление данных

3. **Критичных по времени операций**
   - Когда задержка неприемлема
   - Real-time операции

## Best Practices

### 1. Логирование попыток

```markdown
Перед каждой попыткой логируй:
- Номер попытки
- Причину retry
- Время выполнения

Пример:
"[Retry 2/3] Элемент не найден, повтор через 4s"
```

### 2. Разные лимиты для разных операций

```markdown
Быстрые операции: max 3 попытки, delay 1-2s
Медленные операции: max 5 попыток, delay 2-5s  
Критичные операции: max 2 попытки, quick fail
```

### 3. Graceful degradation

```markdown
Если retry не удался:
1. Логировать детальную ошибку
2. Сохранить screenshot
3. Попробовать альтернативный путь
4. Сообщить пользователю
```

### 4. Условия выхода

```markdown
Определи четкие критерии успеха:
✅ Элемент найден И видим
✅ URL изменился на ожидаемый
✅ Текст содержит ключевое слово

И критерии провала:
❌ Максимум попыток достигнут
❌ Критическая ошибка (404, 403)
❌ Таймаут превышен
```

## Диаграмма процесса

```
Начало
  ↓
Попытка = 1
  ↓
Выполнить операцию
  ↓
Успех? ──Да──> Вернуть результат ──> Конец
  ↓
  Нет
  ↓
Попытка < Max? ──Нет──> Вернуть ошибку ──> Конец
  ↓
  Да
  ↓
Wait (delay * backoff_factor)
  ↓
Попытка += 1
  ↓
(вернуться к "Выполнить операцию")
```
