# Skill: Form Filling

## Описание

Навык автоматического заполнения веб-форм с использованием инструментов браузерной автоматизации.

## Применение

Автоматизация заполнения форм для:
- Регистрации на сайтах
- Оформления заказов
- Отправки заявок
- Заполнения анкет
- Тестирования форм
- Массового ввода данных

---

## Основные техники

### 1. Использование form_input

```markdown
Прямая установка значений в элементы форм по ref_id

## Когда использовать:
- Есть ref_id элемента из read_page
- Нужно быстро заполнить поле
- Работа с текстовыми полями, чекбоксами, селектами

## Синтаксис:
form_input(ref="ref_123", value="текст", tab_id=1)

## Типы значений:
- Текст: value="Иван Иванов"
- Чекбокс: value=true или value=false  
- Селект: value="option_value" или value="Текст опции"
```

### 2. Использование computer + type

```markdown
Клик на поле + набор текста

## Когда использовать:
- Поле видно на скриншоте
- Нужна имитация реального ввода
- Триггер JavaScript событий при вводе

## Последовательность:
1. Click на координаты поля
2. Type текст
3. Опционально: Tab или Enter для перехода

## Пример:
computer([
  {"action": "left_click", "coordinate": [300, 400]},
  {"action": "type", "text": "user@example.com"}
])
```

### 3. Комбинированный подход

```markdown
Использование обоих методов для надежности

## Стратегия:
1. Попробовать form_input (быстрее)
2. Если не сработало -> computer + type
3. Проверить значение после заполнения

## Преимущества:
- Скорость form_input
- Надежность computer
- Fallback при проблемах
```

## Типы полей форм

### Текстовые поля (input type="text")

```markdown
## Техника:
1. Найти поле через read_page или find
2. Использовать form_input с текстом
3. Проверить заполнение

## Особенности:
- Очистка перед вводом (обычно автоматическая)
- Валидация может срабатывать после ввода
- Маски ввода (телефоны, даты)

## Пример:
form_input(ref="ref_email", value="test@example.com")
```

### Email поля (input type="email")

```markdown
## Техника:
Также как текстовые, но со валидацией формата

## Особенности:
- Браузер проверяет формат email
- Использовать корректный email формат
- Может блокировать некорректные значения

## Пример:
form_input(ref="ref_email", value="user@domain.com")
```

### Пароли (input type="password")

```markdown
## Техника:
Как текстовые поля, значение скрыто точками

## Безопасность:
- НЕ хранить пароли в открытом виде
- Получать из безопасного хранилища
- Использовать временные учетные данные для тестов

## Пример:
form_input(ref="ref_password", value="SecurePass123!")
```

### Текстовые области (textarea)

```markdown
## Техника:
Поддерживает многострочный текст

## Особенности:
- Можно вводить переносы строк (\n)
- Часто есть ограничение по символам
- Может быть auto-resize

## Пример:
form_input(ref="ref_comment", value="Строка 1\nСтрока 2\nСтрока 3")
```

### Чекбоксы (input type="checkbox")

```markdown
## Техника:
Использовать булевы значения

## Значения:
- true = установить галочку
- false = снять галочку

## Пример:
form_input(ref="ref_agree", value=true)
form_input(ref="ref_newsletter", value=false)
```

### Радиокнопки (input type="radio")

```markdown
## Техника:
Кликнуть на нужную опцию или использовать form_input

## Особенности:
- Только одна активна в группе
- Использовать value конкретной радиокнопки

## Пример через клик:
computer([{"action": "left_click", "coordinate": [x, y]}])

## Пример через form_input:
form_input(ref="ref_radio_option2", value=true)
```

### Выпадающие списки (select)

```markdown
## Техника:
Установить значение через value атрибут или текст опции

## Варианты:
- По value: form_input(ref="ref_country", value="US")
- По тексту: form_input(ref="ref_country", value="United States")

## Особенности:
- Проверить доступные опции через read_page
- Может быть searchable select (нужен клик + type)

## Пример:
form_input(ref="ref_select", value="option_value")
```

### Файловые загрузки (input type="file")

```markdown
## Ограничение:
НЕ поддерживается автоматическая загрузка файлов

## Причина:
Безопасность браузера запрещает программный доступ к файловой системе

## Альтернатива:
- Попросить пользователя загрузить файл вручную
- Использовать прямые API если доступны
```

### Даты (input type="date")

```markdown
## Техника:
Ввести дату в формате YYYY-MM-DD

## Пример:
form_input(ref="ref_birthdate", value="1990-01-15")

## Альтернатива:
Если не работает, использовать computer + type
```

## Стратегии заполнения форм

### Стратегия 1: Sequential Fill (Последовательное)

```markdown
**Описание**: Заполнять поля одно за другим

1. Получить список всех полей через read_page
2. Для каждого поля:
   a. Определить тип
   b. Подготовить значение
   c. Заполнить через form_input
   d. Опционально: проверить заполнение
3. Нажать кнопку отправки

## Преимущества:
- Простая логика
- Легко отладить
- Понятная последовательность

## Когда использовать:
- Простые формы
- Все поля видимы сразу
- Нет зависимостей между полями
```

### Стратегия 2: Section-by-Section (По секциям)

```markdown
**Описание**: Заполнять форму секциями

1. Идентифицировать секции формы
2. Для каждой секции:
   a. Заполнить все поля секции
   b. Проверить валидацию
   c. Перейти к следующей (клик "Next" или скролл)
3. Финальная отправка

## Преимущества:
- Работа с многостраничными формами
- Валидация по секциям
- Логическая группировка

## Когда использовать:
- Wizard-формы
- Многошаговые процессы
- Длинные формы с группировкой
```

### Стратегия 3: Smart Fill (Умное заполнение)

```markdown
**Описание**: Адаптивное заполнение с проверками

1. Анализ структуры формы
2. Определение обязательных полей
3. Умная валидация:
   - Проверка формата email
   - Проверка длины паролей
   - Валидация дат
4. Заполнение с учетом зависимостей
5. Обработка ошибок валидации

## Преимущества:
- Высокая надежность
- Обработка edge cases
- Адаптация к изменениям

## Когда использовать:
- Сложные формы
- Production окружение
- Критичные операции
```

## Обработка специальных случаев

### Условная логика форм

```markdown
## Проблема:
Некоторые поля появляются/исчезают в зависимости от выбора

## Решение:
1. Заполнить основное поле (trigger)
2. Wait для появления зависимых полей
3. Проверить наличие новых полей
4. Заполнить появившиеся поля

## Пример:
- Выбрать "Юридическое лицо" -> появляются поля ИНН, ОГРН
- Выбрать "Физическое лицо" -> появляются другие поля
```

### CAPTCHA

```markdown
## Проблема:
Нельзя автоматически решить CAPTCHA

## Решения:
1. Использовать тестовое окружение без CAPTCHA
2. Использовать bypass для разработки
3. Попросить пользователя решить вручную
4. Использовать API без веб-интерфейса
```

### Автозаполнение браузера

```markdown
## Проблема:
Браузер может мешать своими предложениями

## Решение:
1. Клик на поле чтобы открыть форму
2. Esc чтобы закрыть предложения
3. Type или form_input значение

## Альтернатива:
Использовать form_input (обычно обходит autocomplete)
```

### Валидация в реальном времени

```markdown
## Проблема:
Форма проверяет поля после каждого ввода

## Решение:
1. Заполнить поле полностью
2. Tab или клик вне поля (trigger blur event)
3. Wait немного для валидации
4. Проверить на ошибки
5. Исправить если нужно

## Пример:
Поле email показывает ошибку пока вводится
Нужно дождаться полного ввода и blur
```

## Best Practices

### 1. Подготовка данных

```markdown
✅ Правильно:
- Подготовить все данные заранее
- Проверить форматы перед заполнением
- Иметь fallback значения
- Валидировать данные на стороне агента

❌ Неправильно:
- Генерировать данные на лету
- Не проверять форматы
- Игнорировать требования полей
```

### 2. Проверка результата

```markdown
- Проверять что поле заполнилось
- Читать сообщения об ошибках
- Проверять итоговое состояние формы
- Скриншот после заполнения для отладки
```

### 3. Обработка ошибок

```markdown
- Логировать проблемы заполнения
- Retry при временных сбоях
- Информировать о невозможности заполнить
- Сохранять состояние для анализа
```

### 4. Производительность

```markdown
- Использовать form_input когда возможно (быстрее)
- Группировать действия computer
- Минимизировать wait между полями
- Заполнять параллельно независимые формы
```

## Примеры сценариев

### Сценарий 1: Простая контактная форма

```markdown
1. Navigate к странице с формой
2. Wait для загрузки
3. read_page для получения ref_id полей
4. Заполнить поля:
   form_input(ref_name, "Иван Иванов")
   form_input(ref_email, "ivan@example.com")
   form_input(ref_message, "Тестовое сообщение")
5. Найти кнопку отправки
6. Click на кнопку
7. Wait и проверить успешную отправку
```

### Сценарий 2: Регистрация пользователя

```markdown
1. Navigate к /register
2. read_page
3. Заполнить последовательно:
   - Имя: form_input(ref_firstname, "Иван")
   - Фамилия: form_input(ref_lastname, "Иванов")
   - Email: form_input(ref_email, "test@example.com")
   - Пароль: form_input(ref_password, "SecureP@ss123")
   - Подтверждение: form_input(ref_confirm, "SecureP@ss123")
   - Согласие: form_input(ref_terms, true)
4. Click "Зарегистрироваться"
5. Проверить успешную регистрацию или ошибки
```

### Сценарий 3: Многошаговая форма заказа

```markdown
Шаг 1: Личные данные
1. Заполнить имя, email, телефон
2. Click "Далее"
3. Wait для загрузки шага 2

Шаг 2: Адрес доставки  
1. Заполнить улицу, город, индекс
2. Выбрать страну из select
3. Click "Далее"
4. Wait для загрузки шага 3

Шаг 3: Оплата
1. Выбрать способ оплаты (radio)
2. Заполнить реквизиты если нужно
3. Click "Оформить заказ"
4. Проверить подтверждение
```

## Граничные случаи

### Динамические формы

```markdown
## Проблема:
Форма загружается через JavaScript

## Решение:
1. Wait после навигации
2. Проверить наличие формы
3. Retry если не загрузилась
4. Использовать wait-strategies pattern
```

### Поля с масками ввода

```markdown
## Проблема:
Телефоны, даты с форматированием

## Решение:
1. Вводить в ожидаемом формате
   - Телефон: "+7 (999) 123-45-67"
   - Дата: "01.01.2024"
2. Или использовать computer + type для имитации ручного ввода
3. Проверить результат после ввода
```

### Скрытые поля

```markdown
## Проблема:
Hidden inputs, honeypot поля

## Решение:
1. Игнорировать hidden поля (обычно)
2. НЕ заполнять honeypot (ловушки для ботов)
3. Заполнять только видимые обязательные поля
```

### Зависимые селекты

```markdown
## Проблема:
Второй select зависит от первого (страна -> город)

## Решение:
1. Выбрать значение в первом select
2. Wait для загрузки опций второго
3. read_page для получения новых опций
4. Выбрать значение во втором select
```

## Инструменты

### Доступные функции:

```markdown
- form_input(ref, value, tab_id)
  Установить значение в поле формы
  
- computer([{"action": "left_click", ...}, {"action": "type", ...}])
  Клик и ввод текста
  
- read_page(tab_id, filter="interactive")
  Получить элементы формы
  
- find(query, tab_id)
  Найти поля по описанию
```
